The theory

So API are the point of connection between your service environment and the outside ecosystem, that can be another service in your domain (back-end or front-end) or someone else service that is trying to connect to yours.

In the specific, we are going to explore a particular kind of API, we call them RESTful endpoints. REST is acronym for REpresentational State Transfer, and it's an architecture to build APIs based on the HTTP protocol and the concept of resources as main actors of the communication.

As you will see soon, every endpoint is defining a way to access a specific resource that is stored or managed by your service. The second key concept of REST is the action against a specific resource. The actions are mapped one to one with the HTTP verbs:

  • GET: is used to read and obtain information on a specific resource
  • POST: translate to the act of creating a new resource
  • PUT - PATCH: the are both used to change the internal status of an existing resource
  • DELETE: (yes you got it) is to remove something.

This is enough for now, there is of course much more to know about RESTful endpoints, so if you are still thirsty for more knowledge follow this link for more interesting concepts.

pokerwars.io engine endpoints

Before you start coding, you need to know which are the key points of connection with pokerwars.io engine so you can send and receive data from your service. We developed a very user friendly architecture, so you just have to know about 3 points of connection:

  • subscribe: this is an endpoint that pokerwars.io is exposing to allow bots to register for a new tournament
  • ping: this is used by the system to check if your bot is alive and ready to play
  • play: the most important one. This is invoked by the system to ask to your bot what's the next action to take in the game
  • notifications: this is not mandatory, but it helps your bot to have extra information on what's going on in the game.

Theory in action

Now, finally, we can have fun. Let's start with creating 2 endpoints: the play and ping ones.

As we figured out before, the play endpoint is invoked by pokerwars.io system sending the game information and it's asking for an action as return so it's a typical POST case.

On the other end, the ping endpoint is more asking for an internal status of the bot, so as we've seen in the theory, it's a GET endpoint.

In the example you can see that we already created the notifications endpoint. We start with the bottle command to identify the type of endpoint @post followed by the URL that we want to expose. In this case the URL starts with the domain name /pokerwars.io and it's followed by the verb connected to the action. 

@post('/pokerwars.io/notifications')

This means that we are exposing a new POST endpoint that external services can invoke to send us notifications.
